# Feature Specification: Extension Scaffold

**Feature Branch**: `001-extension-scaffold`
**Created**: 2026-03-25
**Status**: Draft
**Input**: Initialize the Tube2Chat Chrome extension project using WXT + TypeScript.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Developer can build and load the extension in Chrome (Priority: P1)

A developer clones the repo, installs dependencies, runs the build command, and loads
the resulting `dist/` folder as an unpacked extension in Chrome without errors.

**Why this priority**: If the scaffold doesn't build and load cleanly, no subsequent
feature can be developed or tested. This is the foundational gate.

**Independent Test**: Run `npm run build`, load `dist/` as unpacked extension in Chrome,
verify no errors appear in the Extensions page.

**Acceptance Scenarios**:

1. **Given** a fresh clone of the repo, **When** `npm install && npm run build` is run,
   **Then** a `dist/` folder is produced with a valid `manifest.json` (version 3) and
   no build errors.
2. **Given** the built `dist/` folder, **When** loaded as an unpacked extension in
   Chrome, **Then** the extension appears in the Extensions list with no errors or
   warnings.

---

### User Story 2 — Developer can run the test suite (Priority: P2)

A developer runs the test command and gets a passing result with a placeholder test
confirming the test runner is operational.

**Why this priority**: TDD is a constitution non-negotiable. The test infrastructure must
be ready before any feature code is written.

**Independent Test**: Run `npm test` and observe a passing result (0 failures).

**Acceptance Scenarios**:

1. **Given** a fresh install, **When** `npm test` is run, **Then** Vitest executes
   successfully with no failures.
2. **Given** a placeholder test file, **When** `npm test` is run, **Then** the test
   passes and coverage can be collected.

---

### User Story 3 — Developer can run the extension in dev mode with hot reload (Priority: P3)

A developer runs `npm run dev`, loads the extension in Chrome, and sees changes reflected
without manually rebuilding.

**Why this priority**: Developer experience — not blocking for initial scaffold but
important for iteration speed on subsequent specs.

**Independent Test**: Run `npm run dev`, modify a source file, verify the extension
reloads automatically in Chrome.

**Acceptance Scenarios**:

1. **Given** `npm run dev` is running, **When** a source file is modified and saved,
   **Then** WXT triggers a hot reload visible in the Chrome Extensions page.

---

### Edge Cases

- What if Node.js version is incompatible with WXT? → `.nvmrc` or `engines` field in
  `package.json` documents the required Node version.
- What if `dist/` already exists from a previous build? → Build command cleans and
  regenerates it deterministically.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The project MUST be initialized with WXT as the extension framework.
- **FR-002**: The project MUST use TypeScript as the primary language (strict mode).
- **FR-003**: The manifest MUST declare Manifest Version 3.
- **FR-004**: The project MUST include Vitest as the unit test runner, configured and
  runnable via `npm test`.
- **FR-005**: The folder structure MUST separate source code (`src/`), tests, and build
  output (`dist/` — gitignored).
- **FR-006**: A content script entry point file MUST exist as a stub with zero runtime
  dependencies imported.
- **FR-007**: A service worker entry point MUST exist as a stub (MV3 requires service
  workers, not background pages).
- **FR-008**: The build MUST produce a valid Chrome MV3 extension loadable without errors.
- **FR-009**: A placeholder Vitest test MUST exist and pass to confirm the test
  infrastructure is operational.

### Key Entities

- **manifest.json** (generated by WXT): Declares extension identity, permissions,
  content script URL matches, and service worker. MV3 only.
- **Content script stub**: TypeScript entry point injected into YouTube pages.
  Zero external imports at scaffold stage.
- **Service worker stub**: TypeScript background entry point. Replaces background pages
  (MV3 requirement).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `npm install && npm run build` completes without errors on a fresh clone.
- **SC-002**: The built extension loads in Chrome with zero errors in the Extensions
  management page.
- **SC-003**: `npm test` runs and reports zero failures.
- **SC-004**: The content script entry point contains zero imports from `node_modules`
  at scaffold state.
- **SC-005**: `npm run dev` starts a watch process without errors.

## Assumptions

- Node.js 20+ is available in the developer's environment.
- Chrome (latest stable) is available for manual extension loading.
- No browser extension store submission in this spec — local development only.
- The `dist/` output folder is gitignored and not committed.
- WXT handles `manifest.json` generation from `wxt.config.ts` — no hand-written manifest.
- Content script and service worker are empty stubs at this stage — no business logic.
- The extension will only target `youtube.com` pages (declared in WXT config).
