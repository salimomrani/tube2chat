# Research: Extension Scaffold

**Branch**: `001-extension-scaffold` | **Date**: 2026-03-25

## WXT Project Structure

**Decision**: Use `srcDir: 'src'` in `wxt.config.ts` to keep source separate from config files.

**Rationale**: WXT defaults to root-level entrypoints but supports a cleaner `src/` layout
via the `srcDir` config. Aligns with convention for TypeScript projects.

```
tube2chat/
├── wxt.config.ts           ← always at root (never inside src/)
├── tsconfig.json
├── vitest.config.ts
├── package.json
├── .wxt/                   ← generated by WXT, do not edit
├── dist/                   ← build output, gitignored
└── src/
    ├── entrypoints/
    │   ├── youtube.content/
    │   │   └── index.ts    ← content script (matches youtube.com)
    │   └── background.ts   ← service worker stub
    └── utils/              ← shared helpers (future specs)
```

**Alternatives considered**: Flat root-level entrypoints — rejected because mixing source
files with config at root reduces clarity.

## Content Script Entrypoint Convention

**Decision**: `src/entrypoints/youtube.content/index.ts` using the subdirectory pattern.

**Rationale**: WXT treats every file directly in `entrypoints/` as a separate entrypoint.
Using a subdirectory (`youtube.content/`) allows future addition of `style.css` and
helper files without WXT treating them as new entrypoints. The `matches` are declared
inside the file via `defineContentScript({ matches: ['*://*.youtube.com/*'] })` — WXT
reads them at build time to generate `manifest.json`.

```typescript
// src/entrypoints/youtube.content/index.ts
export default defineContentScript({
  matches: ['*://*.youtube.com/*'],
  runAt: 'document_idle',
  main() {
    // stub — business logic added in spec 002
  },
});
```

**Alternatives considered**: `src/entrypoints/content.ts` (flat file) — rejected because
it prevents adding related files (CSS, helpers) without a structure change later.

## Vitest Integration

**Decision**: Use `WxtVitest()` plugin from `wxt/testing/vitest-plugin` in `vitest.config.ts`.

**Rationale**: WXT ships an official Vitest integration that handles WXT-specific aliases
(`#imports`), extension API polyfills (`fake-browser`), and auto-imports. No third-party
package needed.

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config';
import { WxtVitest } from 'wxt/testing/vitest-plugin';

export default defineConfig({
  plugins: [WxtVitest()],
});
```

For tests using browser APIs: import `fakeBrowser` from `wxt/testing` and call
`fakeBrowser.reset()` in `beforeEach`.

**Alternatives considered**: Vanilla Vitest without WXT plugin — rejected because WXT
path aliases and auto-imports break without the plugin.

## TypeScript Configuration

**Decision**: `tsconfig.json` extends `.wxt/tsconfig.json` (WXT-generated) with `strict: true`.

**Rationale**: `.wxt/tsconfig.json` contains WXT-generated path aliases and type declarations.
Extending it is mandatory — without it, `#imports` and auto-generated types won't resolve.

```json
{
  "extends": "./.wxt/tsconfig.json",
  "compilerOptions": {
    "strict": true,
    "target": "ESNext",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ESNext", "DOM"],
    "skipLibCheck": true
  }
}
```

Note: `wxt prepare` must be run to generate `.wxt/` before TypeScript can resolve types.
The CI/build setup must call `wxt prepare` (or `wxt build` which includes it).

## Manifest V3 Configuration

**Decision**: WXT generates `manifest.json` automatically from entrypoint metadata.
Static overrides (name, version, permissions) go in `wxt.config.ts`.

**Rationale**: WXT targets MV3 by default. The manifest is not hand-written — WXT reads
`matches` from `defineContentScript()` at build time.

```typescript
// wxt.config.ts
export default defineConfig({
  srcDir: 'src',
  manifest: {
    name: 'Tube2Chat',
    version: '0.0.1',
    description: 'Summarize YouTube videos with Gemini',
  },
});
```

**Alternatives considered**: Hand-written `manifest.json` — rejected because WXT's
auto-generation is a core feature; fighting it adds unnecessary maintenance burden.
